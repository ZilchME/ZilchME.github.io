为什么不建议直接 select *

性能问题

  1. 增加不必要的 I/O 和网络开销SELECT * 会读取并返回表中的所有字段，即使你的业务逻辑只需要其中一两个。这会带来多重开销：

    • 磁盘 I/O： 数据库需要从磁盘读取整行数据，如果表中包含 TEXT、BLOB 等大字段，I/O 开销会急剧增加。
    • 网络传输： 大量无用数据通过网络从数据库服务器传输到应用服务器，会占用宝贵的带宽，增加网络延迟。
    • 内存与CPU： 应用层需要为这些多余的数据分配内存，并进行序列化和反序列化，消耗额外的 CPU 资源。

  2. 无法利用覆盖索引 覆盖索引（Covering Index）是一种高效的查询优化方式。当查询所需的所有字段都包含在某个索引中时，数据库可以直接从索引中获取数据，而无需“回表”查询数据行。

    • 示例： 假设 user_info 表有一个 (age, name) 的联合索引。
      • SELECT age, name FROM user_info WHERE age = 18; 可以直接使用覆盖索引，速度很快。
      • SELECT * FROM user_info WHERE age = 18; 由于需要 * 代表的所有字段（如 address），而索引中不包含，数据库必须先用索引找到记录，再回到主表中读取完整数据行，这个过程称为“回表”，会显著降低查询效率。

稳定性与可维护性问题

  1. 表结构变更带来的风险 使用 SELECT * 会使你的代码与数据库表结构产生强耦合。一旦表结构发生变化，比如新增了字段或修改了字段类型，依赖 SELECT * 的代码就可能出现问题。

    • 意外暴露数据： 如果新增了一个包含敏感信息的字段（如 password_hash），SELECT * 会无意中将其暴露给应用层，造成安全风险。
    • 应用崩溃： 如果新增了一个应用层无法识别或处理的字段类型，可能导致程序解析失败甚至崩溃。

  2. 字段顺序依赖问题SELECT * 返回的字段顺序取决于表的物理定义。如果表结构变更导致字段顺序改变，而你的代码是按位置（如 row, row）来获取数据的，就会取到错误的值，引发难以排查的 Bug。

  3. 多表 JOIN 时的列名冲突 在多表连接查询时，SELECT * 会返回所有表的全部字段。如果不同表中有同名字段（如 id, created_at），会导致结果集中出现重复列名，使得应用层在取值时产生混淆或错误。

  4. 代码可读性差 明确的字段列表能让其他开发者一眼看出这个查询到底需要哪些数据。而 SELECT * 隐藏了查询的真实意图，增加了代码理解和后期维护的成本。

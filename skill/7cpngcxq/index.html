如何编写范围明确且与任务匹配的技能

从真实专业知识开始

创建技能时的一个常见陷阱是要求 LLM 生成技能而不提供特定领域的上下文——仅依赖 LLM 的一般训练知识。结果往往是模糊、通用的流程（"适当处理错误"、"遵循身份验证的最佳实践"），而不是使技能有价值的特定 API 模式、边缘情况和项目约定。

有效的技能建立在真实专业知识的基础上。关键在于将特定领域的上下文注入创建过程。

从实际操作任务中提取

与智能体对话完成一个真实任务，提供上下文、纠正和偏好。然后将可重用的模式提取为技能。关注：

  • 有效的步骤 —— 导致成功的操作序列
  • 你做的纠正 —— 你引导智能体方法的地方（例如"使用库 X 而不是 Y"、"检查边缘情况 Z"）
  • 输入/输出格式 —— 数据输入和输出的样子
  • 你提供的上下文 —— 智能体尚不知道的项目特定事实、约定或约束

从现有项目产物中综合

当你拥有大量现有知识时，可以将其输入 LLM 并要求其综合一个技能。从团队实际的事故报告和运行手册中综合的数据管道技能，将优于从通用的"数据工程最佳实践"文章中综合的技能，因为它捕获了你的模式、故障模式和恢复流程。关键在于项目特定的材料，而不是通用参考。

良好的源材料包括：

  • 内部文档、运行手册和风格指南
  • API 规范、模式和配置文件
  • 代码审查评论和问题跟踪器（捕获 recurring 问题和 reviewer 期望）
  • 版本控制历史，特别是补丁和修复（通过实际更改揭示模式）
  • 真实世界的故障案例及其解决方案

通过实际执行来改进

技能的初稿通常需要改进。在真实任务上运行技能，然后将结果——不仅仅是失败的情况——反馈到创建过程中。问：什么触发了误报？遗漏了什么？可以删减什么？

即使是单次执行-然后-修订的过程也能显著提高质量，复杂领域通常需要多次迭代。

提示

阅读智能体的执行追踪，而不仅仅是最终输出。如果智能体在无成效的步骤上浪费时间，常见原因包括：指令太模糊（智能体在找到有效方法前尝试多种方法）、指令不适用于当前任务（智能体仍然遵循）、或者呈现了太多选项而没有明确的默认值。

对于更结构化的迭代方法，包括测试用例、断言和评分，请参阅评估技能输出质量（在新窗口打开）。

明智地使用上下文

一旦技能被激活，其完整的 SKILL.md 内容会连同对话历史、系统上下文和其他活动技能一起加载到智能体的上下文窗口中。技能中的每个 token 都与窗口中的其他所有内容竞争智能体的注意力。

添加智能体缺少的，省略它已知的

专注于智能体没有你的技能就不会知道的内容：项目特定约定、领域特定流程、非显而易见的边缘情况以及要使用的特定工具或 API。你不需要解释 PDF 是什么、HTTP 如何工作或数据库迁移做什么。

<!-- 太冗长 —— 智能体已经知道 PDF 是什么 -->
## 提取 PDF 文本

PDF（便携式文档格式）文件是一种常见的文件格式，包含
文本、图像和其他内容。要从 PDF 提取文本，你需要
使用一个库。推荐使用 pdfplumber，因为它能很好地处理大多数情况。

<!-- 更好 —— 直接进入智能体自己不会知道的内容 -->
## 提取 PDF 文本

使用 pdfplumber 进行文本提取。对于扫描文档，回退到
使用 pdf2image 和 pytesseract。

```python
import pdfplumber

with pdfplumber.open("file.pdf") as pdf:
    text = pdf.pages[0].extract_text()
```

问自己关于每部分内容："没有这个指令，智能体会在这方面出错吗？"如果答案是否定的，就删掉。如果不确定，就测试一下。如果智能体在没有该技能的情况下已经能很好地处理整个任务，那么这个技能可能没有增加价值。参见评估技能输出质量（在新窗口打开）了解如何系统测试。

设计连贯的单元

决定一个技能应该涵盖什么就像决定一个函数应该做什么：你想要它封装一个连贯的工作单元，能够与其他技能良好组合。范围过窄的技能会迫使单个任务加载多个技能，带来开销和指令冲突。范围过宽的技能则难以精确激活。一个用于查询数据库并格式化结果的技能可能是一个连贯的单元，而一个还涵盖数据库管理的技能可能试图做得太多。

追求适度的细节

过于全面的技能可能弊大于利——智能体难以提取相关内容，并可能因为不适用于当前任务的指令而走上无成效的路径。简洁、逐步的指导加上一个工作示例往往比详尽的文档更有效。当你发现自己要覆盖每个边缘情况时，考虑一下大多数情况是否更适合由智能体自己的判断来处理。

用渐进式披露构建大型技能

规范（在新窗口打开）建议将 SKILL.md 保持在 500 行和 5,000 tokens 以内——这只是智能体每次运行所需的核心指令。当一个技能确实需要更多内容时，将详细的参考资料移到 references/ 或类似目录中的单独文件。

关键在于告诉智能体何时加载每个文件。"如果 API 返回非 200 状态码，读取 references/api-errors.md"比通用的"详见 references/"更有用。这让智能体可以按需加载上下文，而不是预先加载，这正是渐进式披露（在新窗口打开）的设计目的。

校准控制力度

并非技能的每个部分都需要相同的指令严格程度。将指令的特定性与任务的脆弱性相匹配。

使特定性与脆弱性匹配

给智能体自由，当多种方法都有效且任务允许变化时。对于灵活的指令，解释原因可能比僵化的指令更有效——理解指令背后目的的智能体能做出更好的上下文相关决策。一个代码审查技能可以描述要查找什么，而不指定确切的步骤：

## 代码审查流程

1. 检查所有数据库查询是否有 SQL 注入（使用参数化查询）
2. 验证每个端点的身份验证检查
3. 查找并发代码路径中的竞态条件
4. 确认错误消息不泄露内部细节

使用严格指令，当操作脆弱、一致性重要或必须遵循特定顺序时：

## 数据库迁移

精确运行此序列：

```bash
python scripts/migrate.py --verify --backup
```

不要修改命令或添加额外的标志。

大多数技能是混合的。独立校准每个部分。

提供默认值，而不是菜单

当多种工具或方法都可能有效时，选择一个默认值并简要提及替代方案，而不是将它们作为同等选项呈现。

<!-- 选项太多 -->
你可以使用 pypdf、pdfplumber、PyMuPDF 或 pdf2image...

<!-- 明确的默认值加上备选方案 -->
对于文本提取，使用 pdfplumber：

```python
import pdfplumber
```

对于需要 OCR 的扫描 PDF，请改用 pdf2image 和 pytesseract。

偏好流程而非声明

技能应该教智能体如何解决一类问题，而不是为特定实例生成什么。比较：

<!-- 特定答案 —— 仅对此确切任务有用 -->
将 `orders` 表与 `customers` 表在 `customer_id` 上连接，筛选
`region = 'EMEA'`，并对 `amount` 列求和。

<!-- 可重用方法 —— 适用于任何分析查询 -->
1. 从 `references/schema.yaml` 读取模式以查找相关表
2. 使用 `_id` 外键约定连接表
3. 将用户请求中的任何筛选条件作为 WHERE 子句应用
4. 根据需要聚合数值列并格式化为 markdown 表格

这并不意味着技能不能包含具体细节——输出格式模板（参见输出格式模板）、约束条件如"绝不输出 PII"和特定工具指令都是有价值的。关键在于，即使个别细节是具体的，方法也应该能够泛化。

有效指令的模式

这些是构建技能内容的可重用技术。并非每个技能都需要所有模式——使用适合你任务的模式。

Gotchas 部分

许多技能中价值最高的内容是一份 gotchas 列表——违背合理假设的环境特定事实。这些不是一般性建议（"适当处理错误"），而是具体的纠正，告诉智能体在未被告知的情况下会犯的错误：

## Gotchas

- `users` 表使用软删除。查询必须包含
  `WHERE deleted_at IS NULL`，否则结果将包括已停用的账户。
- 用户 ID 在数据库中是 `user_id`，在认证服务中是 `uid`，
  在计费 API 中是 `accountId`。这三个值指向同一个东西。
- `/health` 端点只要 Web 服务器在运行就返回 200，
  即使数据库连接已断开。使用 `/ready` 检查完整的服务健康状态。

将 gotchas 保留在 SKILL.md 中，让智能体在遇到情况之前就读到它们。如果告诉智能体何时加载，单独的参考文件也可以工作，但对于非显而易见的问题，智能体可能无法识别触发条件。

提示

当智能体犯了一个你必须纠正的错误时，将纠正添加到 gotchas 部分。这是迭代改进技能最直接的方法之一（参见通过实际执行改进）。

输出格式模板

当你需要智能体以特定格式产生输出时，提供一个模板。这比用散文描述格式更可靠，因为智能体对具体结构的模式匹配效果很好。短模板可以放在 SKILL.md 中；对于较长的模板，或仅在特定情况下需要的模板，将它们存储在 assets/ 中，并在 SKILL.md 中引用，这样它们只在需要时加载。

## 报告结构

使用此模板，根据具体分析调整各部分：

```markdown
# [分析标题]

## 执行摘要
[关键发现的一段概述]

## 主要发现
- 发现 1，附支持数据
- 发现 2，附支持数据

## 建议
1. 具体可行的建议
2. 具体可行的建议
```

多步骤工作流的检查清单

一个明确的检查清单有助于智能体跟踪进度并避免跳过步骤，特别是当步骤有依赖关系或验证关卡时。

## 表单处理工作流

进度：
- [ ] 步骤 1：分析表单（运行 `scripts/analyze_form.py`）
- [ ] 步骤 2：创建字段映射（编辑 `fields.json`）
- [ ] 步骤 3：验证映射（运行 `scripts/validate_fields.py`）
- [ ] 步骤 4：填写表单（运行 `scripts/fill_form.py`）
- [ ] 步骤 5：验证输出（运行 `scripts/verify_output.py`）

验证循环

指示智能体在继续之前验证自己的工作。模式是：完成工作，运行验证器（脚本、参考检查清单或自检），修复任何问题，重复直到验证通过。

## 编辑工作流

1. 进行编辑
2. 运行验证：`python scripts/validate.py output/`
3. 如果验证失败：
   - 检查错误消息
   - 修复问题
   - 再次运行验证
4. 仅在验证通过时继续

参考文档也可以作为"验证器"——指示智能体在最终确定之前根据参考检查其工作。

计划-验证-执行

对于批处理或破坏性操作，让智能体以结构化格式创建中间计划，根据事实来源进行验证，然后才执行。

## PDF 表单填写

1. 提取表单字段：`python scripts/analyze_form.py input.pdf` → `form_fields.json`
   （列出每个字段名称、类型以及是否为必填）
2. 创建 `field_values.json`，将每个字段名称映射到其预期值
3. 验证：`python scripts/validate_fields.py form_fields.json field_values.json`
   （检查每个字段名称是否存在于表单中、类型是否兼容、
   以及必填字段是否缺失）
4. 如果验证失败，修订 `field_values.json` 并重新验证
5. 填写表单：`python scripts/fill_form.py input.pdf field_values.json output.pdf`

关键成分是步骤 3：一个验证脚本，根据事实来源（form_fields.json）检查计划（field_values.json）。像"字段 'signature_date' 未找到 — 可用字段：customer_name、order_total、signature_date_signed"这样的错误给智能体足够的信息来自我纠正。

捆绑可重用脚本

当迭代技能（在新窗口打开）时，比较智能体在不同测试用例中的执行追踪。如果你注意到智能体每次运行都独立地重新发明相同的逻辑——构建图表、解析特定格式、验证输出——这是一个信号，表明应该编写一个经过测试的脚本并捆绑在 scripts/ 中。

关于设计和捆绑脚本的更多信息，请参阅在技能中使用脚本（在新窗口打开）。
